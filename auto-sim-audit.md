# Технический аудит сайта auto-sim.ru

Дата проверки: 25 июня 2026  
Сайт: https://auto-sim.ru/

## Краткий вывод

Сайт выглядит как одностраничное React-приложение, собранное без серверного рендера и без нормальной SEO-инфраструктуры. В текущем виде это скорее интерактивная визитка, чем посадочная страница, рассчитанная на органический трафик, локальное SEO и масштабирование.

Главная проблема не в том, что сайт "на React". Проблема в том, что он отдает поисковому роботу почти пустой HTML, а весь контент появляется только после загрузки и выполнения JavaScript. Для лендинга локального бизнеса это слабое техническое решение: хуже индексируется, сложнее расширяется, хуже контролируются метаданные, хуже работают социальные превью и служебные страницы.

Рекомендация: переделать сайт на Next.js с server-side rendering / static generation, нормальными маршрутами, метаданными, sitemap, robots.txt, schema.org и оптимизацией изображений.

## Что проверено

- Открытие сайта в браузере.
- Серверный HTML главной страницы.
- Заголовки ответа сервера.
- Наличие robots.txt и sitemap.xml.
- Метаданные страницы.
- Заголовочная структура после рендера.
- Ссылки и изображения.
- Lighthouse-аудит локально: mobile и desktop.
- Наличие structured data / schema.org в бандле.

## Критичные аргументы

### 1. Сервер отдает почти пустой HTML

Фактический HTML главной страницы весит 524 байта и содержит только:

```html
<div id="root"></div>
```

Контент сайта, секции, цены, отзывы, статьи, контакты и CTA появляются только после загрузки JavaScript-бандла.

Почему это плохо:

- поисковик сначала получает почти пустую страницу;
- SEO зависит от того, как и когда робот выполнит JavaScript;
- контент хуже доступен для быстрых парсеров, соцсетей, мессенджеров, некоторых ботов и AI-поиска;
- сайт нельзя считать полноценным SEO-лендингом;
- любое падение JS означает пустую страницу для пользователя.

Для коммерческого лендинга это технически слабая база. Такие страницы лучше делать через Next.js: контент должен приходить в HTML сразу.

### 2. Нет нормального robots.txt

Адрес `https://auto-sim.ru/robots.txt` возвращает не robots.txt, а HTML главной страницы.

Lighthouse подтвердил проблему: `robots.txt is not valid`, найдено 14 ошибок.

Почему это плохо:

- поисковые роботы не получают корректные инструкции;
- невозможно явно указать sitemap;
- это выглядит как неправильная настройка сервера;
- техническое качество сайта для SEO ниже базового уровня.

### 3. Нет нормального sitemap.xml

Адрес `https://auto-sim.ru/sitemap.xml` также возвращает HTML главной страницы.

Почему это плохо:

- поисковики не получают карту сайта;
- новые разделы, статьи, акции и страницы хуже обнаруживаются;
- нельзя нормально управлять индексацией;
- для локального бизнеса это потеря простого SEO-инструмента.

### 4. SPA-fallback настроен слишком грубо

Сервер возвращает `200 OK` и HTML приложения для служебных и несуществующих URL:

- `/robots.txt`
- `/sitemap.xml`
- `/manifest.json`
- `/site.webmanifest`
- `/privacy`
- `/privacy/`

Почему это плохо:

- несуществующие или служебные страницы выглядят как валидные HTML-страницы;
- поисковик может видеть мусорные URL;
- нет нормальной 404-логики;
- это признак шаблонного деплоя SPA без продуманной серверной конфигурации.

### 5. Нет meta description

В HTML нет `<meta name="description">`.

Lighthouse: `Document does not have a meta description`.

Почему это плохо:

- поисковик сам формирует сниппет как получится;
- ниже контроль над CTR в выдаче;
- для локальных запросов вроде "автосимулятор Оренбург", "гонки на симуляторе Оренбург", "VR автосимулятор" страница не помогает сама себе.

### 6. Ошибка в title

Текущий title:

```text
АвтоСим | Клуб Автосимуляторов в Орнебурге
```

Вместо "Оренбурге" указано "Орнебурге".

Почему это плохо:

- ошибка в ключевом локальном слове;
- снижает доверие пользователя;
- ухудшает локальную релевантность;
- для бизнеса в Оренбурге это особенно неприятный дефект.

### 7. Нет canonical

На странице не найден `rel="canonical"`.

Почему это плохо:

- нет явного канонического URL;
- при появлении дублей, UTM-меток и разных URL-версий поисковику сложнее понять основную страницу;
- для одностраничного сайта это легко исправляется, но сейчас не сделано.

### 8. Нет structured data / schema.org

В бандле и HTML не обнаружены признаки JSON-LD:

- `LocalBusiness`
- `Organization`
- `FAQPage`
- `application/ld+json`

Почему это плохо:

- сайт не объясняет поисковикам, что это локальный бизнес;
- не размечены адрес, телефон, рейтинг, FAQ, соцсети, организация;
- хуже готовность к расширенным сниппетам и AI-поиску;
- для локального клуба это один из самых очевидных quick wins, но его нет.

### 9. Слабая мобильная производительность

Локальный Lighthouse mobile:

- Performance: 71/100
- SEO: 83/100
- First Contentful Paint: 3.6 s
- Largest Contentful Paint: 4.8 s
- Speed Index: 5.8 s
- Time to Interactive: 6.4 s
- Unused JavaScript: экономия около 100 KiB

Почему это плохо:

- мобильный пользователь долго ждет первого нормального контента;
- органический трафик чаще мобильный;
- тяжелая JS-архитектура для лендинга не оправдана;
- часть JS не используется на старте, но все равно грузится.

### 10. Сайт не готов к нормальному расширению

На странице визуально есть блок "Полезные статьи", но технически это не отдельные индексируемые страницы. В текущей архитектуре это контент внутри SPA, а не полноценная SEO-структура.

Почему это плохо:

- нельзя нормально продвигать статьи отдельно;
- нельзя собирать long-tail запросы;
- нельзя строить нормальную структуру: услуги, режимы, дни рождения, корпоративы, FAQ, блог;
- весь сайт фактически сведен к одному URL.

## Что нужно сделать

1. Перенести сайт на Next.js.
2. Отдавать основной контент в HTML через SSR/SSG.
3. Сделать нормальные маршруты:
   - `/`
   - `/prices`
   - `/birthday`
   - `/corporate`
   - `/faq`
   - `/blog/...`
   - `/privacy`
4. Добавить корректные `robots.txt` и `sitemap.xml`.
5. Исправить title и добавить meta description.
6. Добавить canonical.
7. Добавить JSON-LD:
   - LocalBusiness
   - Organization
   - FAQPage
   - BreadcrumbList
8. Настроить 404 и убрать SPA-fallback для служебных файлов.
9. Оптимизировать загрузку изображений через Next Image или аналогичный пайплайн.
10. Разнести контент по нормальным страницам, чтобы сайт мог ранжироваться не только по бренду.

## Формулировка для заказчика

Текущий сайт технически сделан как SPA на React: сервер отдает почти пустой HTML, а весь контент появляется только после выполнения JavaScript. Для современного SEO-лендинга локального бизнеса это плохая база. Сайт может выглядеть нормально в браузере, но для поисковых систем, социальных превью, AI-поиска и технической индексации он подготовлен слабо.

Отдельно обнаружены базовые SEO-ошибки: отсутствуют корректные robots.txt и sitemap.xml, нет meta description, нет canonical, нет structured data для локального бизнеса, есть ошибка в title с названием города. Служебные URL возвращают HTML главной страницы с кодом 200, что говорит о неправильной серверной настройке.

Вывод: сайт лучше не "допиливать точечно", а пересобрать на Next.js с серверным рендером, нормальными страницами, SEO-метаданными, картой сайта, schema.org и корректной серверной конфигурацией. Это даст нормальную техническую основу для продвижения, рекламы, аналитики и дальнейшего развития.
